## 2. TypeScript тАФ Strict Rules

### 2.1 Compiler & Lint Configuration
- `strict: true` must be enabled in every `tsconfig.json`.
- `noImplicitAny: true` тАФ the `any` type is **banned** project-wide.
  - If a type is genuinely unknown at compile time, use `unknown` and narrow with type guards.
- `noUncheckedIndexedAccess: true` тАФ array/object index access must be safely narrowed.
- Enable `strictNullChecks`, `strictFunctionTypes`, `strictBindCallApply`.

### 2.2 Type Safety
- Every function must have **explicit return types**.
- Every public API boundary (route handler, service function, repository method) must use a **branded/opaque type or Zod schema** for its inputs and outputs.
- **No type assertions** (`as`) unless accompanied by a comment justifying why it is safe.
- Prefer `interface` for object shapes that will be extended; use `type` for unions, intersections, and computed types.

### 2.3 Payload Validation
- All incoming HTTP payloads (bin telemetry, user input, webhook data) **must** be validated at the API boundary using **Zod** (or equivalent runtime schema validator).
- Validated payloads must be transformed into a **branded type** before being passed deeper into the application.
- Validation failures must return structured error responses (`{ error: string; details: ZodIssue[] }`), never raw stack traces.

### 2.4 Telemetry Payload Schema
The canonical incoming bin payload:
```ts
// Validated at the ingestion endpoint тАФ never trust raw input
interface RawBinTelemetry {
  bin_id: string;
  timestamp: string; // ISO 8601
  temperature_c: number;
  gas_ppm: number;
  weight_kg: number;
  battery_level: number; // integer 0тАУ100
}
```

### 2.5 Error Handling
- Use **Result types** (`Result<T, E>`) or **discriminated unions** instead of throwing exceptions for expected error paths.
- Unexpected errors may throw but must be caught by a top-level error-handling middleware.
- Never silently swallow errors. Every `catch` must log or re-throw.

## 3. Supabase & PostgreSQL тАФ Database Rules

### 3.1 Supabase as Backend Platform
- **Supabase** is the primary backend platform. Use the Supabase client library (`@supabase/supabase-js`) for all DB operations.
- Use the **Supabase MCP server** during development for schema management, migrations, SQL execution, and type generation.
- Use **Supabase Edge Functions** for serverless backend logic where appropriate (e.g., webhook handlers, scheduled jobs).
- Use **Supabase Realtime** for live dashboard updates (bin status changes, new alerts).
- Enable **Row Level Security (RLS)** on all tables. Every table must have RLS policies enforcing organization-scoped access.

### 3.2 Schema Management
- All schema changes must go through **Supabase migrations** (via the MCP `apply_migration` tool or Supabase CLI).
- **Never** run ad-hoc `ALTER TABLE` statements against shared environments.
- Every migration must have a corresponding **down/rollback** migration.

### 3.3 Naming Conventions
- Tables: `snake_case`, plural (e.g., `organizations`, `bin_telemetry_readings`).
- Columns: `snake_case` (e.g., `created_at`, `battery_level`).
- Primary keys: always `id` (UUID, generated by the database via `gen_random_uuid()`).
- Foreign keys: `<referenced_table_singular>_id` (e.g., `organization_id`, `bin_id`).
- Indexes: `idx_<table>_<column(s)>` (e.g., `idx_telemetry_bin_id_timestamp`).

### 3.4 Data Integrity
- All foreign keys must have explicit `ON DELETE` behavior (typically `CASCADE` for child data, `RESTRICT` for critical references).
- Use `CHECK` constraints for domain rules (e.g., `battery_level BETWEEN 0 AND 100`).
- Use `NOT NULL` by default; nullable columns require justification in a comment.
- Timestamps: always store in UTC. Use `TIMESTAMPTZ` for all time columns.

### 3.5 Query Patterns
- Use the **Supabase client** for standard CRUD operations.
- Use **parameterized queries** exclusively for any raw SQL. String interpolation in SQL is forbidden.
- Use Supabase's built-in PostgREST API for simple queries; reserve raw SQL for complex analytical work.
- Time-series queries against `bin_telemetry_readings` should leverage composite indexes on `(bin_id, recorded_at)`.

### 3.6 Row Level Security (RLS)
- **Every table** must have RLS enabled.
- Policies must enforce that users can only access rows belonging to their `organization_id`.
- Bin telemetry ingestion (from IoT devices) uses a **service role key** to bypass RLS.
- Dashboard queries use the **anon/authenticated role** with RLS enforced.

## 4. Functional Programming Patterns тАФ Backend Logic

### 4.1 Pure Functions First
- Business logic (freshness scoring, donation eligibility, alert routing) must be implemented as **pure functions**: no side effects, no mutation, deterministic output.
- Side effects (database writes, HTTP calls, sending alerts) must be pushed to the **edges** of the call stack (handlers/controllers), not embedded in core logic.

### 4.2 Immutability
- Use `readonly` and `ReadonlyArray<T>` for all data flowing through business logic.
- Prefer `Object.freeze`, spread operators, or immutable-update patterns over mutation.
- Never mutate function arguments.

### 4.3 Composition over Inheritance
- No classes for business logic. Use **plain functions** composed via pipes or sequential application.
- Middleware and route handlers should be composable higher-order functions, not class methods.
- If a "service" is needed, represent it as a **module** exporting pure functions that accept explicit dependencies.

### 4.4 Pipeline Pattern for Telemetry Processing
Structure the telemetry ingestion flow as a composable pipeline:
```
validate тЖТ enrich тЖТ score тЖТ decide тЖТ persist тЖТ notify
```
Each stage is a pure function accepting the output of the previous stage.

## 5. API Design

- RESTful endpoints. Resource-oriented URLs: `/api/v1/bins/:binId/telemetry`.
- All endpoints must be versioned under `/api/v1/`.
- Use standard HTTP status codes: `201` for creation, `400` for validation errors, `401`/`403` for auth, `404` for missing resources, `500` for unexpected errors.
- Paginate list endpoints using cursor-based pagination (`?cursor=<id>&limit=20`).

## 6. Authentication & Authorization

### 6.1 Auth0 тАФ Primary Identity Provider
- Use **Auth0** for all authentication across the platform (frontend and backend).
- Frontend: Use the **Auth0 Next.js SDK** (`@auth0/nextjs-auth0`) for session management.
- Backend API: Validate **Auth0 JWTs** on every request using `jwks-rsa` + `express-jwt` or equivalent.
- Store Auth0 `sub` (user ID) in the `users` table as `auth0_id` for mapping Auth0 identities to local user records.

### 6.2 Per-Bin API Key Authentication (IoT)
- Each registered bin is issued a **unique API key** stored in the `bins` table (`api_key_hash` column).
- Bins send the API key in the `Authorization: Bearer <key>` header with every telemetry POST.
- The ingestion endpoint validates the key, identifies the bin, and rejects requests with invalid/revoked keys.
- API keys are generated on bin registration and can be **rotated** or **revoked** per bin.

### 6.3 Role-Based Access Control (RBAC)
- Implement RBAC via Auth0 **roles and permissions** (configured in Auth0 dashboard).
- Roles: `admin`, `store_manager`, `food_bank_coordinator`.
- Permissions are embedded in the Auth0 JWT and checked by middleware.
- Organization-scoped data access: enforce at the **RLS policy level** (Supabase) and in middleware.

## 7. Testing

- Minimum coverage target: **80%** for business logic modules, **60%** for API routes.
- Use **Vitest** as the test runner.
- Business logic (pure functions) тЖТ **unit tests** (fast, no DB).
- API routes тЖТ **integration tests** (use a test database, reset between suites).
- Frontend тЖТ **React Testing Library** for components; **Playwright** for E2E.
- Never commit tests that depend on external services without mocks/stubs.

## 10. Environment & Secrets

- All secrets and connection strings go in `.env` files (never committed).
- Use a `.env.example` with placeholder values for onboarding.
- Validate all environment variables at startup using a Zod schema. Crash immediately on missing or invalid config.
- Required env vars include: `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `AUTH0_DOMAIN`, `AUTH0_CLIENT_ID`, `AUTH0_CLIENT_SECRET`, `AUTH0_AUDIENCE`.

## 11. Documentation тАФ Mandatory Maintenance

### 11.1 Documentation Lives in `docs/`
The `docs/` directory at the project root is the **single source of truth** for project documentation. It must be kept up-to-date as code evolves.

### 11.2 Required Documents
Every one of these files **must exist** and be updated when the corresponding area changes:

| Document | Purpose | Update When |
|---|---|---|
| `architecture.md` | System architecture, component diagram, tech stack decisions | Any structural change |
| `data-flow.md` | End-to-end data flow from bin POST тЖТ DB тЖТ alert тЖТ food bank | Pipeline or routing changes |
| `api-reference.md` | All API endpoints with request/response schemas | Any endpoint added/modified |
| `onboarding.md` | Setup guide for new developers (env, tools, first run) | Tooling or setup changes |
| `mock-contracts.md` | Contracts for mock functions (freshness scoring, donation routing) | Mock interface changes |
| `database.md` | Table descriptions, relationships, migration history | Any schema migration |

### 11.3 Documentation Rules
- **Every PR that changes a public interface** (API endpoint, DB schema, mock contract) **must include a docs update**.
- Documentation must include **concrete examples** (sample payloads, curl commands, expected responses).
- Use **mermaid diagrams** for architectural and data-flow documentation.
- Keep language clear and jargon-free so that any team member (including the freshness scoring and donation routing teams) can understand the system.
